# Tuning methodology

Beluga separates **correctness** from **strength tuning**. Nothing in the search
or eval is tuned by guesswork that isn't gated by a statistical test.

## 1. Search-parameter tuning (SPSA via self-play)

Reduction/pruning constants (LMR base/divisor, null-move R, futility/RFP margins,
LMP counts, aspiration delta) are tuned with **SPSA** under self-play:

1. Expose a constant as a tunable.
2. Run paired games (e.g. cutechess-cli) perturbing `+c`/`-c`.
3. Step the parameter along the estimated gradient.
4. **Confirm every change with SPRT** (`scripts/sprt.sh`) before keeping it.

Bounds used for SPRT regression gates: `elo0 = 0`, `elo1 = 3`,
`alpha = beta = 0.05` for incremental changes; widen for risky rewrites.

## 2. Evaluation tuning (Texel)

The handcrafted eval terms (material, PSTs, mobility, passed pawns, king safety)
are tuned by the **Texel method**: minimize the prediction error of the static
eval against game results over a large quiet-position set.

```
loss = mean( (sigmoid(K * eval(fen)) - result)^2 )
```

- `result ∈ {0, 0.5, 1}` from the side-to-move perspective.
- `K` is fitted first, then parameters by gradient/coordinate descent.
- Positions must be **quiet** (no side in check, no winning capture available) to
  avoid teaching the eval tactics the search should find.

`engine-tune` is the harness scaffold for this; feed it labeled EPD/FEN data
generated by `scripts/`.

## 3. Data generation

`scripts/gen_training_data.py` plays fast self-play games and emits
`(fen, search_eval, game_result)` records — the same format used both for Texel
tuning of the handcrafted eval and for training an NNUE network (Phase 5).

## 4. Discipline

- One change at a time; SPRT each.
- Keep the previous binary as the baseline; never tune against an untested build.
- Use identical time controls and hardware for both sides of a match.
- Re-run the regression suite (`scripts/run_tests.sh`) before any tuning run so a
  correctness bug is never mistaken for an Elo change.
